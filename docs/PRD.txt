# PRD

## 1. Overview
A lightweight, mobile-first Personal Finance Tracker for Indonesian students and early-career professionals. Backend: Laravel REST API (Sanctum/Passport). Frontend: React + Vite + Tailwind. Deploy to Oracle Free Tier VPS (Ubuntu). Focus: fast mobile data entry, CSV import, monthly budgets & alerts, localized IDR/Bahasa Indonesia, PDF/Excel exports, and on-device / self-hosted AI-powered insights (no external AI APIs).

Goals for Phase 1 (1 semester):
- Core ledger: transactions, categories, accounts (cash/e-wallet).
- Monthly budgets with alerts and simple projection.
- CSV import, fast-entry UI, dashboard charts, monthly report exports (PDF/Excel).
- Conversational AI insights using a local/embedded model or lightweight time-series library (no third‑party AI APIs).
- Authentication (Sanctum/Passport), privacy-first defaults.

Success metrics:
- Daily active users (DAU) vs target class size.
- Time-to-entry: <10s for common transaction.
- CSV import: 95% success on test files.
- Reports generation under 30s (queued for large exports).

## 2. Requirements
Functional
- Secure user auth (email/password, tokens).
- CRUD for transactions, categories, accounts, budgets.
- Monthly budget creation, tracking, alerts (in-app + optional email/push).
- CSV import with validation & chunked processing.
- Exports: PDF and Excel per month.
- Dashboard endpoints for charts (monthly totals, category breakdown).
- AI insights endpoint: forecast, anomaly detection, saving recommendations (local).
- Reports generation via queued job (e.g., GenerateReport job).

Non-functional
- Mobile-first responsive UI; low bandwidth friendly.
- Deployable on single small VPS (Oracle Free Tier).
- Data privacy: user data stored only on VPS, encrypted at rest/transport.
- Localized currency formatting (IDR) and Bahasa Indonesia strings.
- Reasonable performance for single-tenant/small multi-tenant usage.

Constraints (see section 7).

## 3. Core Features
1. Fast Transaction Entry
   - Quick-add modal with smart defaults (last category, recurring amounts).
   - Bulk CSV import with mapping UI.

2. Transaction Management & Categories
   - Multi-category tagging (one primary + optional tags).
   - Accounts (Cash, E-wallets) with balances.

3. Budgets & Alerts
   - Monthly budgets by category; progress %; threshold alerts (50/75/90%).
   - In-app alerts; optional email/push via SMTP/FCM.

4. Dashboard & Reports
   - Summary card (balance, monthly income/expense).
   - Charts: category breakdown, cashflow forecast, weekly trend.
   - Monthly PDF/Excel exports (via queued job).

5. AI-Powered Insights (Local)
   - Forecast next-month spend per category.
   - Anomaly detection for outliers.
   - Concrete saving recommendations (e.g., cut X in dining to meet goal).
   - Conversational UI on the frontend that queries the API for analysis.

6. Import/Export
   - CSV import with mapping, validation, chunked processing.
   - Exports via Laravel Excel / DomPDF.

7. Settings & Onboarding
   - Currency set to IDR, language default Bahasa Indonesia.
   - Data export/download and account deletion.

## 4. User Flow
1. Onboarding
   - Sign up (email + password) → basic profile (monthly income, default account).
   - Option: import CSV (map columns) or skip.

2. Daily Use
   - Fast-add transaction → select account, amount, category, date, note.
   - Dashboard shows updated totals and progress vs budgets.

3. Budgeting
   - Create monthly budget per category → receive alerts as thresholds cross.
   - View budget breakdown and projected shortfalls.

4. Reports & Export
   - Request monthly report → backend enqueues GenerateReport job → user notified when ready to download (PDF/Excel).

5. AI Insights
   - User opens "Analyze" → API returns forecasts, anomalies, and tailored recommendations in Bahasa Indonesia and IDR formatting.
   - Option to ask follow-up questions in conversational UI (handled client-side + API).

6. Data Management
   - Import CSV page for bulk uploads; background processing with progress/status.
   - Settings page for email/push preferences and data export/delete.

## 5. Architecture & Integrations
High level:
- Frontend: React + Vite + Tailwind + TanStack Query.
- Backend: Laravel REST API (Sanctum or Passport), queued jobs (Redis or database queue), scheduled tasks (cron).
- DB: MySQL/MariaDB.
- Exports: maatwebsite/excel, DomPDF.
- CSV import: Laravel file validation + chunked processing (Storage::disk('local')->putFileAs()).
- AI: Embedded/local service (PHP ML lib or small Python microservice on same VPS) or TensorFlow.js on frontend.
- Notifications: Laravel Notifications (in-app); optional SMTP / FCM / Resend as integrations.

## 6. Database Schema
All timestamps (created_at, updated_at). Use foreign keys where appropriate.

Tables:

- users
  - id (PK, bigint)
  - name varchar(191)
  - email varchar(191) unique
  - password varchar(255)
  - locale varchar(10) default 'id'
  - currency varchar(5) default 'IDR'
  - settings json nullable
  - created_at, updated_at

- accounts
  - id, user_id FK -> users
  - name varchar(100)
  - type enum('cash','ewallet','bank','other')
  - starting_balance decimal(14,2)
  - current_balance decimal(14,2)
  - metadata json
  - created_at, updated_at

- categories
  - id, user_id nullable (null = default shared)
  - name varchar(100)
  - type enum('expense','income','transfer')
  - color varchar(7)
  - created_at, updated_at

- transactions
  - id, user_id FK
  - account_id FK
  - type enum('expense','income','transfer')
  - amount decimal(14,2)
  - date date
  - category_id FK nullable
  - note text nullable
  - meta json (tags, source)
  - imported boolean default false
  - external_id varchar(191) nullable
  - soft deletes (deleted_at)
  - created_at, updated_at
  - indexes: (user_id,date), (account_id)

- budgets
  - id, user_id FK
  - month char(7) e.g., '2026-05'
  - category_id FK
  - amount decimal(14,2)
  - spent decimal(14,2) cached
  - created_at, updated_at
  - unique(user_id, month, category_id)

- budget_alerts
  - id, budget_id FK
  - thresholds json e.g., [50,75,90]
  - channels json e.g., ['in_app','email']
  - last_alerted json (timestamp per threshold)
  - created_at, updated_at

- imports
  - id, user_id FK
  - filename varchar(255)
  - status enum('uploaded','mapping','processing','completed','failed')
  - rows_total int
  - rows_processed int
  - errors json nullable
  - created_at, updated_at

- reports
  - id, user_id FK
  - month char(7)
  - format enum('pdf','xlsx')
  - status enum('queued','processing','ready','failed')
  - file_path varchar(255) nullable
  - created_at, updated_at

- ai_analyses
  - id, user_id FK
  - scope varchar(50)
  - input json
  - result json
  - created_at, updated_at

- notifications
  - id, user_id FK
  - type varchar(100)
  - data json
  - read_at datetime nullable
  - created_at

Indexes, FK constraints, and cascading rules should be enforced for data integrity.

## 7. Constraints
- Hosting: Single small VPS (Oracle Free Tier) — limited CPU/RAM → prefer DB on same machine, database tuning, prefer database queue over Redis if resources limited.
- Storage: Keep exports and imports limited; purge old files; limit synchronous exports to small sizes.
- Compute: Heavy ML tasks should be lightweight or run as batch jobs; consider a small Python microservice only if PHP options are insufficient.
- No external bank integrations or payment processors in Phase 1 (explicitly excluded).
- No external AI APIs (OpenAI/Groq) — use local models/libraries only as per project scope.
- Privacy/legal: Data stays on VPS; implement export and delete flows to satisfy user control.
- Timeline: Fit within one semester; scope features to MVP first (core ledger, budgets, CSV import, exports, simple AI insights).
- Team & skills: Laravel required (backend), React + Vite (frontend), basic DevOps for Ubuntu/Nginx deploy.
- Libraries noted: maatwebsite/excel, DomPDF, phpml/phpml or a small Python service for Prophet, Sanctum/Passport for auth.
- Optional integrations can be added later (SMTP, FCM). Example external email API (if used later): POST /emails with JSON {"from": "...", "to": [...], "subject": "...", "html": "..."}.

Endpoints and payload formats above incorporate the provided API DOCS guidance: CSV import via Laravel Storage and chunked processing, queued job GenerateReport for exports, Laravel Notifications for budget alerts, and use of Laravel Sanctum/Passport for authentication. No external APIs are required for Phase 1.

BRD — Phase 2 (Extension of Phase 1 PRD)

Purpose
- Expand the MVP from Phase 1 into a more robust, user-centric Personal Finance Tracker for Indonesian students and early-career professionals while preserving privacy-first, single-VPS deployability and no external AI APIs.
- Deliver higher automation, better offline/online sync, richer AI insights (still local), and pragmatic integrations (SMTP/FCM) to increase retention and time-savings.

Goals (90-day Phase 2)
- Improve user retention & utility: reduce manual entry, surface actionable recommendations, and enable basic sharing/backup.
- Add recurring transactions, device sync & conflict resolution, attachments (receipt images), scheduled jobs for ML training & heavier analysis, and export management/purging.
- Harden ops: automated backups, monitoring, lightweight CI, and optimized VPS footprint.

Success metrics
- DAU retention rate +10% vs Phase 1 baseline.
- Time-to-entry reduced by 25% via rules/recurring.
- CSV import success rate stays >=95% with expanded parser library support.
- AI analysis job turnaround <=60s for typical datasets (queued for heavy analyses).
- Exports automatically purge after retention policy; storage saved by 30%.

Scope (what Phase 2 adds)
- Recurring transactions: create, edit, auto-generate instances.
- Auto-mapping rules: remember user CSV mapping and infer mappings; bank CSV presets.
- Attachments: upload receipt images, store encrypted on-disk; optional OCR microservice (local) to suggest fields.
- Device sync: register devices, client-side change queue, server-side simple conflict resolution by timestamp + user choice.
- Offline-first support (client): local cache + eventual sync endpoints.
- AI enhancements (local): scheduled retraining jobs, improved forecasts (category & user-level), confidence scores, scenario simulations ("If I cut dining by 20%") — still via local Python microservice or lightweight PHP libs.
- Webhooks: optional internal webhooks for report-ready, import-complete, budget-alerts (for advanced users).
- Admin / maintenance endpoints: purge exports, view queue length, re-enqueue job.
- Improved notification channels: SMTP (Resend-compatible) and FCM for push.
- Multi-device session management and revocation.

Out of scope (Phase 2)
- Direct bank integrations (OAuth, PSD2), external commercial AI APIs, multi-tenant SaaS billing, complex role-based access control.

Stakeholders
- Product Owner: student/early-career user community representatives
- Backend Lead (Laravel), Frontend Lead (React + Vite), DevOps (Ubuntu/Nginx), Data/ML Engineer (Python optional)
- QA, Localization lead (Bahasa Indonesia/IDR formatting), Security reviewer

Risks & Mitigations
- VPS resource exhaustion from ML/OCR: schedule heavy jobs, CPU/IO limits, fallback to queued batch, provide “run later” UI.
- Data loss: automated daily DB dumps + encrypted offsite copy (SFTP) + user export flow.
- Privacy: encryption at rest (file system encryption + DB column encryption for sensitive metadata) and strict access controls.

MVP Deliverables (Phase 2 priority order)
1. Recurring transactions + rules engine (CSV mapping persistence).
2. Attachments + basic local OCR job (queue).
3. Device registration + sync endpoints; conflict resolution UIs.
4. AI microservice enhancements: scheduled retrain, scenario analysis, confidence.
5. Admin endpoints (purge/requeue) + export retention policy.
6. SMTP/FCM integration & webhook support.

Detailed Tech Stack (Phase 2)

Core
- Backend: Laravel 10 LTS (PHP 8.2+), Sanctum for SPA tokens (Passport optional for OAuth tokens).
- Frontend: React 18 + Vite, TailwindCSS, TanStack Query (data fetching & cache), Zustand or React Query mutations for offline queue.
- DB: MySQL 8 / MariaDB 10.6 (single-node on VPS). Use InnoDB, tuned buffers for limited RAM.
- Queue: Database queue (default) for constrained VPS; Redis optional if provisioned (faster).
- Web server: Nginx + PHP-FPM.
- Files: OS filesystem (encrypted LUKS or fs-level encryption) under /var/www/storage, limited retention; optional SFTP offsite backup.
- Export libraries: maatwebsite/excel, DomPDF, Spatie/laravel-activitylog.
- CSV processing: laravel-chunk-upload + chunked reader, custom parsers and presets.
- AI & ML:
  - Option A (preferred Phase 2): small Python microservice (Flask/FastAPI) deployed as systemd service or container on same VPS; libraries: Prophet (fbprophet via prophet), scikit-learn, numpy, pandas, statsmodels — used for forecasting & anomaly detection.
  - Option B (fallback): phpml/phpml + simple time-series heuristics; lower accuracy but single-language.
  - OCR (optional): Tesseract (local binary) invoked by queued job for receipt parsing.
- Attachments: store encrypted files; use Laravel Filesystem with local disk and encryption layer (Laravel encryption of metadata/paths).
- Monitoring & logs: Prometheus node exporter (minimal) or lightweight monitors (Netdata), centralized logs via rotated files + optional external shipper.
- CI/CD: GitHub Actions for linting, tests, builds; deploy script: rsync build artifacts + artisan migrate --force + composer install.
- TLS: Let's Encrypt (certbot). Fail2ban, UFW hardening.
- Backups: daily mysqldump + gz + encrypted offsite (SFTP) retention 14 days; exports retention policy (e.g., 30 days).
- Security: HTTPS, DB user with least privilege, password hashing via Argon2id/Bcrypt, rate-limits on auth endpoints, Sentry (optional) for error monitoring.

Deployment Topology (single VPS)
- Nginx reverse proxy -> PHP-FPM -> Laravel app
- MySQL on same host
- Python microservice runs on localhost:5000 (systemd) communicating via HTTP + local unix socket
- Worker: php artisan queue:work (database) or supervisor configured
- Cron: laravel schedule via crontab
- Storage: /var/www/app/storage (encrypted)
- Ports: 80/443 open; 22 for admin (key-based)

Operational guidelines
- Keep heavy jobs queued; schedule heavy ML retrains during night hours.
- Limit synchronous exports; route large exports to background GenerateReport job.
- Monitor disk usage and enforce purge policies automatically.
